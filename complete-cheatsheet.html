Java Fundamentals

Everything from data types to exception handling — covering every concept interviewers test in 2–3 year experience rounds.

🔄
Mutable vs Immutable
Most asked basic concept in Java interviews
Easy▾
📖 Definition
Mutable objects can be modified after creation — their state (fields) can change. Immutable objects cannot be changed once created — any "modification" returns a new object. Immutable objects are thread-safe by default.
🌍 Real-world Example
Mutable: A Google Doc — you can edit it anytime. Immutable: A printed book — to "change" it, you print a new edition. In Java: String is immutable, StringBuilder is mutable.
MutableVsImmutable.java
// ✅ IMMUTABLE — String
String s = "Hello";
s.concat(" World");   // creates a NEW String — s is still "Hello"
System.out.println(s); // Hello (unchanged!)
s = s.concat(" World"); // now s points to new object

// 🔧 MUTABLE — StringBuilder
StringBuilder sb = new StringBuilder("Hello");
sb.append(" World");    // SAME object modified
System.out.println(sb); // Hello World

// ✅ How to create your own Immutable class
public final class Money {              // final — can't extend
    private final double amount;         // final — can't reassign
    private final String currency;
    public Money(double amount, String currency) {
        this.amount = amount;
        this.currency = currency;
    }
    public double getAmount()   { return amount; }
    public String getCurrency() { return currency; }
    // No setters! Returns new object for "changes"
    public Money add(double extra) {
        return new Money(amount + extra, currency);
    }
}
🎯 Interview Questions
QWhy is String immutable in Java?Medium▾
✅ Answer
String is immutable because: 1. Security — usernames, passwords, URLs stored as strings can't be changed mid-flight. 2. Thread-safety — shared across threads without sync. 3. String Pool — JVM can reuse strings in pool only if they can't change. 4. HashCode caching — HashMap keys need a stable hashCode.
💡 Say this: "If String were mutable, a HashMap key could change after insertion, making the entry unrecoverable."
QRules to make a class immutable in Java?Medium▾
✅ Answer
1. Declare class as final (no subclassing). 2. All fields private final. 3. No setters. 4. Initialize all fields via constructor. 5. For mutable fields (List, Date): return defensive copies in getters. Java 14+ Records are immutable by default.

String vs StringBuilder vs StringBuffer
Performance difference — classic interview trap
Easy▾
📖 Definition
String is immutable — every concat creates a new object in heap. StringBuilder is mutable and NOT thread-safe — best for single-threaded use. StringBuffer is mutable and thread-safe (synchronized) — slightly slower due to locks.
🌍 Real-world Example
Building a large JSON/CSV response in a loop — always use StringBuilder. Using String + in a loop creates thousands of objects, killing performance. For Ford's report generation via Java REST APIs, StringBuilder would be essential for string assembly.
StringPerformance.java
// ❌ BAD — String in loop (1000 new objects!)
String result = "";
for (int i = 0; i < 1000; i++) result += i; // slow!

// ✅ GOOD — StringBuilder (single object)
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) sb.append(i);
String result = sb.toString();

// Thread-safe — StringBuffer
StringBuffer sbuf = new StringBuffer();
// synchronized — safe for multi-threaded writing
🎯 Interview Questions
QWhen would you use StringBuffer over StringBuilder?Medium▾
✅ Answer
Use StringBuffer only when multiple threads will append to the same object concurrently. In 99% of cases, StringBuilder is correct and faster. If thread-safety is needed at a higher level (e.g., the whole method is synchronized), still use StringBuilder.
💡 Real answer: "In modern Java, StringBuffer is rarely used. We handle thread safety at a higher level."
⚖️
== vs .equals() vs hashCode()
Asked in 90% of Java interviews
Easy▾
📖 Definition
== compares references (memory addresses) for objects — checks if two variables point to the exact same object. .equals() compares content/value — can be overridden to define what "equal" means. hashCode() returns an integer representing the object — equal objects MUST have the same hashCode.
EqualsDemo.java
String a = new String("hello");
String b = new String("hello");
a == b           // false — different heap objects
a.equals(b)    // true  — same content

// String Pool (literals)
String x = "hello";
String y = "hello";
x == y           // true  — same pool reference

// Always override both equals AND hashCode together
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof User)) return false;
    User u = (User) o;
    return Objects.equals(id, u.id);
}
@Override
public int hashCode() { return Objects.hash(id); }
🎯 Interview Questions
QWhat happens if you override equals but not hashCode?Hard▾
✅ Answer
The HashMap contract breaks. If two objects are equal (equals returns true) but have different hashCodes, they'll be placed in different buckets. map.get(key) will compute hashCode, go to the wrong bucket, and return null — even though the key "exists".
💡 Rule: "If a.equals(b), then a.hashCode() == b.hashCode(). ALWAYS." Use @Override to ensure you don't forget.

Checked vs Unchecked Exceptions
Compile-time vs runtime — know the difference cold
Easy▾
📖 Definition
Checked exceptions extend Exception — must be declared with throws or caught at compile time (IOException, SQLException). Unchecked exceptions extend RuntimeException — not required to catch, discovered at runtime (NullPointerException, ArrayIndexOutOfBoundsException).
🌍 Real-world Example
Checked: Reading a file — the file might not exist, compiler forces you to handle it. Unchecked: NullPointerException — a bug in your code, not a recoverable condition.
ExceptionDemo.java
// Checked — must handle
try (FileReader fr = new FileReader("data.txt")) {
    // auto-closed via try-with-resources
} catch (IOException e) {
    log.error("File read failed", e);
    throw new ServiceException("Cannot read config", e); // wrap
}

// Custom exception hierarchy
public class ResourceNotFoundException extends RuntimeException {
    public ResourceNotFoundException(String resource, Long id) {
        super(resource + " not found with id: " + id);
    }
}
// Usage in Spring Boot service
return userRepo.findById(id)
    .orElseThrow(() -> new ResourceNotFoundException("User", id));
🎯 Interview Questions
QWhat is the difference between throw and throws?Easy▾
✅ Answer
throw is used inside a method to actually throw an exception object. throws is used in the method signature to declare that a method might throw a checked exception — it's a warning to callers.
void readFile() throws IOException { // declares throw new IOException("File not found"); // actually throws }
QDoes finally always execute?Medium▾
✅ Answer
Yes, almost always. Exceptions: (1) System.exit() called, (2) JVM crashes, (3) Thread is killed. Even if try or catch throws an exception, finally runs. If both catch and finally throw, the finally exception overwrites the catch exception.

HashMap Internal Working
Most popular intermediate question — know it deeply
Hard▾
📖 Definition
HashMap stores key-value pairs in an array of buckets. Key's hashCode() determines the bucket index. Collisions (same bucket, different keys) form a LinkedList; Java 8+ converts to a Red-Black Tree when bucket size exceeds 8 — giving O(log n) instead of O(n) worst case.
🌍 Real-world Example
A library card catalogue — drawers are buckets (indexed by first letter of book name). Most drawers have a short list. If one drawer overflows, the librarian switches to a tree index inside that drawer for faster search.
HashMapInternal.java
// Default capacity=16, load factor=0.75
// Rehash when 75% full → doubles capacity
Map<String,Integer> map = new HashMap<>(32, 0.75f); // pre-size if known

// put() flow internally:
// 1. hash = key.hashCode() ^ (h >>> 16)
// 2. index = hash & (capacity-1)
// 3. bucket[index] exists? → check equals()
// 4. no? add new Node. yes? update value
// 5. bucket size > 8? → treeify to Red-Black Tree

// Thread-safe alternatives
Map<String,Integer> safe1 = Collections.synchronizedMap(new HashMap<>());
Map<String,Integer> safe2 = new ConcurrentHashMap<>(); // better!
// ConcurrentHashMap uses segment locking (Java 7) / CAS (Java 8)
🎯 Interview Questions
QWhat is the difference between HashMap, LinkedHashMap, TreeMap, ConcurrentHashMap?Medium▾
✅ Answer
HashMap: O(1) get/put, no ordering, allows null key/values. LinkedHashMap: maintains insertion order (or access order). TreeMap: keys sorted (natural order or Comparator), O(log n). ConcurrentHashMap: thread-safe, no null keys/values, uses CAS (compare-and-swap) in Java 8+.
💡 Use LinkedHashMap for LRU cache. Use TreeMap for range queries. Use ConcurrentHashMap in multi-threaded environments.
QWhy can't we use a mutable object as a HashMap key?Hard▾
✅ Answer
If you mutate the key after putting it in the map, its hashCode changes. The entry is now in the wrong bucket. When you try to get it, HashMap looks in a different bucket and returns null — the entry is "lost" even though it's in the map.
📦
Autoboxing, Unboxing & Wrapper Classes
Know the NullPointerException traps
Medium▾
📖 Definition
Autoboxing: automatic conversion of primitive to Wrapper class (int → Integer) by the compiler. Unboxing: Wrapper → primitive (Integer → int). Wrapper classes (Integer, Double, Boolean…) add OOP capabilities to primitives — allow null, usable in collections.
🌍 Real-world Example
When you add an int to a List<Integer>, the compiler silently autoboxes it. When you do int x = integerObj, it unboxes. The danger: if the Integer is null, unboxing throws NullPointerException.
AutoboxingDemo.java
List<Integer> list = new ArrayList<>();
list.add(42);           // autoboxing: 42 → Integer.valueOf(42)
int val = list.get(0); // unboxing: Integer → int

// ⚠️ Integer cache: -128 to 127 are cached
Integer a = 127;  Integer b = 127;
a == b    // true (same cached object)
Integer c = 128;  Integer d = 128;
c == d    // false (new objects!) → always use .equals()

// ⚠️ NPE trap
Integer score = null;
if (score > 0) {} // NullPointerException! unboxing null
// Fix:
if (score != null && score > 0) {}
🎯 Interview Questions
QWhat is Integer cache and why does Integer == fail above 127?Hard▾
✅ Answer
Java caches Integer objects from -128 to 127 in a pool (IntegerCache). Within that range, Integer.valueOf() returns the same cached object. Beyond 127, new Integer objects are created — so == compares different references and returns false. Always use .equals() for Integer comparisons.

Java Memory Model — Stack, Heap, GC
JVM internals — senior-level questions
Hard▾
📖 Definition
Stack: stores method frames + local variables + references. Per-thread, LIFO, auto-cleaned. Fast. Heap: stores all objects. Shared across threads. Managed by Garbage Collector (GC). Metaspace (Java 8+): stores class metadata (replaced PermGen). GC: automatically reclaims unreachable heap objects.
MemoryModel.java
public static void main(String[] args) {
    int age = 25;          // Stack: primitive value
    String name = "Sri";    // Stack: reference → Heap: String object
    User u = new User();  // Stack: reference → Heap: User object
}  // method returns → stack frame popped, age/name/u refs gone
   // Heap User object is now unreachable → eligible for GC

// GC Roots: static vars, local vars in active stacks, JNI refs
// G1GC (default Java 9+): concurrent, low-pause, region-based
// ZGC (Java 15+): sub-millisecond pauses for large heaps

// Tuning flags
// -Xms512m   initial heap
// -Xmx2g     max heap
// -XX:+UseG1GC  use G1 collector
🎯 Interview Questions
QWhat causes a memory leak in Java?Hard▾
✅ Answer
Memory leaks happen when objects are reachable but never used. Common causes: (1) Static collections that grow indefinitely. (2) Listeners/callbacks never de-registered. (3) Long-lived objects holding short-lived references. (4) ThreadLocal variables not cleaned up. (5) Unclosed resources (Connection, Stream). Tools: JVisualVM, Eclipse MAT, async-profiler.

Design Principles

SOLID, DRY, KISS, YAGNI — the foundations every senior dev is expected to know and apply.

📐
SOLID Principles
5 principles that make code maintainable and scalable
Medium▾
📖 Definition
SOLID is an acronym for 5 design principles: Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion. They guide writing code that is easy to extend, test, and maintain without breaking existing functionality.
🌍 Real-world Example
SOLID principles are what Spring Boot is built on. S — UserService only handles users, not emails. O — Add new payment type without editing PaymentService. L — Every Repository subtype works where JpaRepository is expected. I — ReadRepository separate from WriteRepository. D — Controller depends on UserService interface, not the impl.
SOLIDDemo.java
// S — Single Responsibility
// ❌ Bad: UserService handles user logic AND sends emails
// ✅ Good: UserService for users, EmailService for emails

// O — Open/Closed (open to extend, closed to modify)
interface Discount { double apply(double price); }
class FlatDiscount   implements Discount { ... } // extend, don't modify
class PercentDiscount implements Discount { ... }

// L — Liskov Substitution
// Every subtype must be replaceable for its parent without breaking things
Animal a = new Dog(); // Dog must behave as Animal expects

// I — Interface Segregation
interface Readable  { User findById(Long id); }   // split!
interface Writable  { User save(User u); }

// D — Dependency Inversion
// ❌ Bad
class UserController { private UserServiceImpl svc = new UserServiceImpl(); }
// ✅ Good — depends on interface, Spring injects impl
class UserController {
    private final UserService svc; // interface!
    public UserController(UserService svc) { this.svc = svc; }
}
🎯 Interview Questions
QHow have you applied SOLID in your project?Medium▾
✅ Answer (personalized for your Ford project)
"In Ford Pro, I applied S by separating report generation from report delivery — one service assembled the data, another handled GCP Cloud Storage upload and Pub/Sub notification. D was achieved through Spring's DI — controllers depend on service interfaces, making unit testing with Mockito easy. O was used for our metric report types — each new report type extended a base interface without touching existing code."
💡 Always tie SOLID back to your real project. Generic answers don't impress — specific ones do.

DRY — Don't Repeat Yourself
Eliminate duplication with abstraction
Easy▾
📖 Definition
Every piece of knowledge should have a single, unambiguous representation in a system. When you copy-paste logic, a bug fix in one place is missed in the others. DRY is about abstraction — extract repeated logic into a function, class, or constant.
🌍 Real-world Example
If you validate an email in 5 different controllers instead of a shared EmailValidator utility, that's DRY violation. When the email regex changes, you update 5 places and miss 2 — bugs appear.
DRY.java
// ❌ Violation — same logic duplicated
class UserController   { if(!email.contains("@")) throw ...; }
class AdminController  { if(!email.contains("@")) throw ...; }
class SignupController { if(!email.contains("@")) throw ...; }

// ✅ DRY — single source of truth
class EmailValidator {
    public static void validate(String email) {
        if (!email.matches("^[\\w.-]+@[\\w.-]+\\.[a-z]{2,}$"))
            throw new InvalidEmailException(email);
    }
}
// Spring Boot way: use @Valid + custom @ValidEmail annotation
💋
KISS & YAGNI
Keep It Simple + You Aren't Gonna Need It
Easy▾
📖 Definition
KISS: Keep It Simple, Stupid — prefer the simplest solution that works. Complex code is harder to debug, test, and maintain. YAGNI: You Aren't Gonna Need It — don't build features "just in case". Build what is needed now; add complexity when requirements actually arrive.
🌍 Real-world Example
KISS violation: A junior dev builds a plugin system for a report generator that has 2 report types — over-engineering. YAGNI violation: Adding multi-currency support before a single customer asks for it. In Agile, YAGNI aligns with "just enough" — build for current sprint's requirements.
🎯 Interview Questions
QHow do you balance YAGNI with good architecture?Hard▾
✅ Answer
Use interfaces and abstraction where change is likely (e.g., storage layer, payment gateway) but don't implement multiple strategies until needed. The abstraction costs almost nothing — the implementation costs time. "Make it work, make it right, then make it fast." Follow the Rule of Three: refactor to a general solution only after the third occurrence of similar code.
💡 Say: "I follow YAGNI by writing only what's needed for the current story, but I use interfaces at integration boundaries so future extensions don't require rewriting existing code."

Design Patterns — Singleton, Factory, Builder, Strategy
GoF patterns used daily in Spring Boot
Medium▾
📖 Definition
Design patterns are proven, reusable solutions to commonly occurring problems. Not specific code — rather templates for how to structure your classes. Categorized as Creational (how to create), Structural (how to compose), Behavioral (how to interact).
Patterns.java
// SINGLETON — one instance (Spring beans are singleton by default)
public enum AppConfig { INSTANCE; // best singleton in Java
    public String getDb() { return System.getenv("DB_URL"); }
}

// FACTORY — create object without exposing creation logic
class NotificationFactory {
    public static Notification create(String type) {
        return switch (type) {
            case "EMAIL" -> new EmailNotification();
            case "SMS"   -> new SmsNotification();
            default -> throw new IllegalArgumentException(type);
        };
    }
}

// BUILDER — complex object construction (Lombok @Builder)
User user = User.builder()
    .name("Srinivasan")
    .email("sri@ford.com")
    .role(Role.DEVELOPER)
    .build();

// STRATEGY — swap algorithms at runtime
interface SortStrategy { void sort(List<Integer> list); }
class Sorter {
    private SortStrategy strategy;
    public void setStrategy(SortStrategy s) { strategy = s; }
    public void sort(List<Integer> l) { strategy.sort(l); }
}
🎯 Interview Questions
QWhat design patterns does Spring Boot use internally?Medium▾
✅ Answer
Singleton: beans are singleton-scoped. Factory: BeanFactory/ApplicationContext creates beans. Proxy: @Transactional, @Cacheable, @Async use AOP proxies. Template Method: JdbcTemplate, RestTemplate. Observer: ApplicationEvent + @EventListener. Decorator: HttpServletRequestWrapper. Strategy: MessageConverter, HandlerMapping.
🔮
CAP Theorem & Eventual Consistency
Distributed systems — senior level
Hard▾
📖 Definition
CAP Theorem: a distributed system can guarantee only 2 of 3 — Consistency (all nodes see same data), Availability (every request gets a response), Partition Tolerance (system works despite network failures). Since Partition is unavoidable in real networks, you choose CA or CP.
🌍 Real-world Example
CP (Consistency + Partition): Bank transactions — correctness over availability. HBase, ZooKeeper. AP (Availability + Partition): Shopping cart, DNS, social media feeds — availability over consistency. DynamoDB, Cassandra. Eventual consistency: data will be consistent eventually, but not immediately.
🎯 Interview Questions
QWhy can't we have CAP — all three?Hard▾
✅ Answer
During a network partition, nodes can't communicate. If you still serve requests (Available), different nodes will have different data (not Consistent). If you wait for consensus (Consistent), you can't serve requests (not Available). So you MUST sacrifice one when partition occurs.

Caching Strategies

Redis, cache invalidation, TTL, eviction policies — asked in every system design and backend interview.

⚡
What is Caching?
The most powerful performance optimization
Easy▾
📖 Definition
Caching stores frequently accessed data in fast memory (RAM) to avoid slow operations (DB queries, API calls). A cache hit returns data instantly; a cache miss fetches from the source. It's the most impactful performance optimization in backend systems.
🌍 Real-world Example
YouTube thumbnails are cached in CDN servers near you — not fetched from Google's datacentre every time. In Ford Pro, report data that doesn't change could be cached in Redis so repeated requests don't hit Firestore every time, reducing cost and latency.
SpringCaching.java
// Spring Boot Cache — @Cacheable
@Service
public class UserService {

    @Cacheable(value = "users", key = "#id")
    public User findById(Long id) {
        // called only on CACHE MISS — result stored in cache
        return userRepo.findById(id).orElseThrow();
    }

    @CacheEvict(value = "users", key = "#id")
    public void deleteUser(Long id) {
        userRepo.deleteById(id); // also removes from cache
    }

    @CachePut(value = "users", key = "#result.id")
    public User update(User user) {
        return userRepo.save(user); // updates cache too
    }
}

// Redis config — application.yml
// spring.cache.type=redis
// spring.redis.host=localhost
// spring.redis.port=6379
// spring.cache.redis.time-to-live=600000  (10 min TTL)
🎯 Interview Questions
QWhat is cache invalidation and why is it hard?Medium▾
✅ Answer
Cache invalidation is removing stale data from the cache when the source data changes. It's considered one of the two hardest problems in CS (along with naming things). Challenges: (1) knowing WHEN data changed, (2) propagating invalidation to all cache nodes in distributed systems, (3) avoiding thundering herd (all requests hitting DB simultaneously after cache cleared).
💡 Common strategies: TTL (time-to-live — expire after N seconds), event-driven invalidation (Kafka event triggers cache clear), write-through (update cache and DB simultaneously).
QCache-aside vs Write-through vs Write-back — explain eachHard▾
✅ Answer
Cache-aside (Lazy): app checks cache first, if miss fetches from DB and populates cache. Most common. Risk: cold start. Write-through: every write updates both cache and DB simultaneously. Always consistent but slower writes. Write-back (Write-behind): write to cache immediately, async write to DB later. Fastest writes but data loss risk if cache fails before DB sync.
QWhat are cache eviction policies?Medium▾
✅ Answer
LRU (Least Recently Used): evict item not accessed for longest time. Most common — used in Redis. LFU (Least Frequently Used): evict least accessed item. FIFO: evict oldest item. TTL: evict after time expires regardless of usage. Redis default is LRU. Use LFU when access frequency matters more than recency.
🔴
Redis — Advanced Patterns
Distributed cache, pub/sub, rate limiting
Medium▾
📖 Definition
Redis is an in-memory data store supporting Strings, Lists, Sets, Sorted Sets, Hashes, Streams. Used as a cache, session store, pub/sub message broker, distributed lock, rate limiter, and leaderboard. Data persists via RDB snapshots or AOF (Append Only File).
RedisPatterns.java
// Rate limiting with Redis
public boolean isAllowed(String userId) {
    String key = "rate:" + userId;
    Long count = redisTemplate.opsForValue().increment(key);
    if (count == 1) redisTemplate.expire(key, 60, TimeUnit.SECONDS);
    return count <= 100; // 100 req/min
}

// Distributed lock
boolean locked = redisTemplate.opsForValue()
    .setIfAbsent("lock:resource", "1", 30, TimeUnit.SECONDS);
if (!locked) throw new LockException("Resource busy");

// Session store
// spring.session.store-type=redis
// → sessions stored in Redis, works across multiple app instances
🎯 Interview Questions
QHow does Redis handle persistence — RDB vs AOF?Hard▾
✅ Answer
RDB (Redis Database): periodic snapshots (e.g., every 5 min). Fast restore, but may lose last 5 min data on crash. AOF (Append Only File): logs every write command. More durable (can replay from log), but larger files and slower. Hybrid: use both — AOF for durability, RDB for fast restart. For cache-only Redis, no persistence needed.
🛡️
Cache Stampede & Thundering Herd
Production problems every senior must know
Hard▾
📖 Definition
Cache stampede (thundering herd): when a popular cache key expires, hundreds of requests simultaneously hit the database to rebuild it — overloading the DB. Solutions: probabilistic early expiration, mutex locking (only one request rebuilds, others wait), or staggered TTLs.
StampedeProtection.java
// Mutex pattern — only one thread rebuilds cache
public Data getData(String key) {
    Data cached = redis.get(key);
    if (cached != null) return cached;

    String lockKey = "lock:" + key;
    boolean lock = redis.setIfAbsent(lockKey, "1", 10, TimeUnit.SECONDS);
    if (lock) {
        try {
            Data fresh = db.fetch(key);   // only ONE thread does this
            redis.set(key, fresh, 300, TimeUnit.SECONDS);
            return fresh;
        } finally { redis.delete(lockKey); }
    } else {
        Thread.sleep(100);  // others wait briefly
        return getData(key); // retry — cache should be warm
    }
}
System Design

Load Balancing

How traffic is distributed across servers — algorithms, types, health checks, and sticky sessions.

⚖️
What is Load Balancing?
Scale horizontally, eliminate single point of failure
Easy▾
📖 Definition
Load balancing distributes incoming requests across multiple servers to prevent overload, ensure high availability, and improve throughput. It eliminates single points of failure. Can operate at Layer 4 (TCP/IP) or Layer 7 (HTTP/application) of the OSI model.
🌍 Real-world Example
When you access google.com, your request doesn't go to one server — it hits a load balancer that routes you to one of thousands of servers. If one server crashes, the load balancer stops sending traffic to it (health check fails) and routes to healthy servers.
LoadBalancingAlgorithms.txt
// ALGORITHMS

Round Robin: requests go 1→2→3→1→2→3 sequentially
  Best for: homogeneous servers, stateless apps

Weighted Round Robin: powerful servers get more requests
  Server A (8 core) = weight 4, Server B (4 core) = weight 2

Least Connections: route to server with fewest active connections
  Best for: long-lived requests (file uploads, WebSockets)

IP Hash: same client IP always → same server (sticky)
  Best for: stateful sessions without Redis

Random: pick random server. Simple, works well at scale

Consistent Hashing: used in distributed caches (Redis Cluster)
  Minimises remapping when nodes added/removed
🎯 Interview Questions
QWhat is a sticky session and when would you use it?Medium▾
✅ Answer
Sticky session (session affinity) routes a user's requests to the same server every time, using a cookie or IP hash. Use it when server-side state (in-memory session) can't be moved to a shared store. Downside: if that server crashes, session is lost; uneven load distribution. Better approach: store sessions in Redis so any server can handle any request.
💡 "At Ford, we used Spring Session with Redis — sessions are centrally stored, allowing any of our instances to serve any request. No sticky sessions needed."
QL4 vs L7 Load Balancer — difference?Hard▾
✅ Answer
L4 (Transport layer): routes based on IP and port only — can't inspect HTTP content. Faster, simpler. AWS NLB (Network Load Balancer). L7 (Application layer): can inspect HTTP headers, URL paths, cookies. Enables path-based routing (/api → service1, /admin → service2), SSL termination, rate limiting. AWS ALB, Nginx, Spring Cloud Gateway. In microservices, API Gateway is an L7 LB.
🏥
Health Checks & Circuit Breaker
How load balancers detect and handle failures
Medium▾
📖 Definition
Health checks: load balancer periodically pings each server (e.g., GET /actuator/health). If a server fails N checks, it's removed from rotation. Circuit Breaker: when a downstream service is failing, stop sending requests immediately (open circuit) and return a fallback — prevents cascading failures.
CircuitBreaker.java (Resilience4j)
@Service

🎯 Interview Questions
QWhat is the difference between Circuit Breaker, Retry, and Timeout?Medium▾
✅ Answer
Timeout: fail fast if response takes too long — prevents threads being stuck forever. Retry: automatically retry failed requests N times with backoff — handles transient failures. Circuit Breaker: after N failures, stop trying entirely and use fallback — prevents flooding a broken service. Use all three together (Resilience4j supports this).

Consistent Hashing
How distributed caches scale without reshuffling everything
Hard▾
📖 Definition
Consistent hashing places both servers and keys on a virtual ring. A key maps to the nearest clockwise server. When a server is added/removed, only keys in that portion of the ring re-map — minimizing redistribution (~1/N of keys, not all).
🌍 Real-world Example
Redis Cluster uses consistent hashing with 16,384 hash slots. When you add a new Redis node, only some slots (and their keys) migrate — not the whole dataset. Used in DynamoDB, Cassandra, Memcached, Nginx upstreams.
🎯 Interview Questions
QWhy use consistent hashing instead of modulo hashing?Hard▾
✅ Answer
With modulo hashing (key % N servers), adding a server changes N — almost every key remaps. With a 10-server cluster adding one server, 90% of cached keys would need to move, causing a massive cache miss storm. Consistent hashing limits redistribution to ~1/N keys — adding 1 server to a 10-server cluster moves only ~10% of keys.

📨
What is a Message Queue?
Decouple services, handle spikes, ensure delivery
Easy▾
📖 Definition
A message queue enables asynchronous communication between services. Producer sends a message to the queue; consumer reads and processes it independently. This decouples services (producer doesn't wait for consumer), handles traffic spikes (queue absorbs bursts), and ensures reliable delivery even if consumer is temporarily down.
🌍 Real-world Example
When you place an Amazon order: the Order Service immediately returns "Order placed!" and sends an event to a queue. The Inventory Service, Email Service, and Shipping Service each consume the event independently and asynchronously. User gets instant response; complex processing happens in the background.
📨 Kafka
Distributed log — retain messages for days/weeks
Millions of messages/sec throughput
Consumer tracks its own offset (position)
Best for event streaming, analytics, audit log
Messages not deleted after consumption
🐰 RabbitMQ
Traditional message broker — push model
Complex routing (exchanges, bindings)
Messages deleted after ACK from consumer
Best for task queues, RPC patterns
Easier to set up for simple use cases

🎯 Interview Questions
QWhat is at-least-once vs at-most-once vs exactly-once delivery?Hard▾
✅ Answer
At-most-once: message delivered 0 or 1 time — fire and forget, may lose messages. At-least-once: message delivered 1+ times — no loss, but duplicates possible. Consumer must be idempotent (processing same message twice has same effect as once). Exactly-once: guaranteed once — hardest, Kafka supports it with transactions + idempotent producers (high overhead).
💡 "In Ford's Pub/Sub pipeline, we used at-least-once delivery with idempotent consumers — checking if the report was already generated before re-generating."
QWhat is Kafka's consumer group and partition?Medium▾
✅ Answer
A partition is an ordered, immutable sequence of messages in a topic. Messages within a partition maintain order. A consumer group: multiple consumers sharing the workload — each partition is consumed by exactly one consumer in a group. Adding consumers to a group = horizontal scaling. Messages are replayed by setting offset back — great for bug fixes and re-processing.

Dead Letter Queue & Poison Messages
What happens when a consumer can't process a message?
Medium▾
📖 Definition
A Dead Letter Queue (DLQ) is where messages go after failing processing N times. A poison message is one that always fails (e.g., bad data format, bug). Without DLQ, poison messages block the queue forever. DLQ lets you isolate failures, alert on them, and reprocess after fixing the bug.

Outbox Pattern & Saga
Distributed transaction patterns without 2PC
Hard▾
📖 Definition
Outbox Pattern: instead of publishing to Kafka directly (which may fail separately from DB), write the event to an outbox table in the same DB transaction as your business data. A separate relay reads the outbox and publishes to Kafka. Guarantees atomicity. Saga: breaks a distributed transaction into local transactions with compensating rollbacks.
🎯 Interview Questions
QWhy not use 2PC (two-phase commit) in microservices?Hard▾
✅ Answer
2PC requires all participants to lock resources until the coordinator confirms. In microservices: (1) a slow/failed coordinator blocks all services, (2) it's synchronous — kills throughput, (3) doesn't work across different database technologies. The Saga pattern with compensation is preferred — each service does its work locally and compensates on failure, without distributed locks.
SQL & Databases

SQL Deep Dive

Joins, indexes, window functions, transactions, query optimization — every question from basic SELECT to advanced analytics.

🔗
SQL JOINs — All Types
Most tested SQL topic in every interview
Easy▾
📖 Definition
JOINs combine rows from two or more tables based on a related column. INNER JOIN: only matching rows. LEFT JOIN: all rows from left + matches from right (null if no match). RIGHT JOIN: reverse. FULL OUTER JOIN: all rows from both, nulls where no match. CROSS JOIN: every combination (Cartesian product).
joins.sql
-- INNER JOIN: users who have orders
SELECT u.name, o.total
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- LEFT JOIN: ALL users, even those without orders
SELECT u.name, COUNT(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name;

-- SELF JOIN: find employees and their managers
SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

-- Find users with NO orders (LEFT JOIN trick)
SELECT u.name
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;
🎯 Interview Questions
QWhat is the difference between WHERE and HAVING?Easy▾
✅ Answer
WHERE: filters rows BEFORE grouping (before GROUP BY). Cannot use aggregate functions. HAVING: filters AFTER grouping (after GROUP BY). Can use aggregate functions like COUNT, SUM.
-- Users with more than 5 orders SELECT user_id, COUNT(*) AS cnt FROM orders WHERE status = 'COMPLETED' -- filter rows first GROUP BY user_id HAVING COUNT(*) > 5; -- then filter groups
QWhat is the difference between DELETE, TRUNCATE, and DROP?Easy▾
✅ Answer
DELETE: removes rows with WHERE condition, can rollback, fires triggers, logs each row. TRUNCATE: removes ALL rows, cannot rollback (DDL), no triggers, resets identity/auto-increment, much faster. DROP: removes the entire table structure + data permanently.
📋
Indexes — How & When to Use
The most impactful database performance tool
Medium▾
📖 Definition
An index is a data structure (B-Tree by default in PostgreSQL) that speeds up data retrieval. Like a book's index — instead of scanning every page (full table scan), jump directly to the page. Indexes speed up reads but slow down writes (index must be updated on every INSERT/UPDATE/DELETE).
indexes.sql
-- Basic index
CREATE INDEX idx_users_email ON users(email);

-- Composite index (order matters! email+status != status+email)
CREATE INDEX idx_orders_user_status ON orders(user_id, status);

-- Partial index (only index active users)
CREATE INDEX idx_active_users ON users(email) WHERE active = true;

-- Unique index
CREATE UNIQUE INDEX idx_users_email_unique ON users(email);

-- EXPLAIN ANALYZE — check if index is being used
EXPLAIN ANALYZE
SELECT * FROM users WHERE email = 'test@ford.com';
-- Look for "Index Scan" (good) vs "Seq Scan" (no index used)
🎯 Interview Questions
QWhen should you NOT add an index?Medium▾
✅ Answer
(1) Small tables — full scan is faster than index lookup. (2) Low cardinality columns — indexing a boolean (only 2 values) is useless, DB still scans 50% of rows. (3) Frequently written tables — every INSERT/UPDATE updates the index. (4) Columns never used in WHERE/JOIN/ORDER BY. Rule: add indexes when you see slow queries via EXPLAIN ANALYZE, not preemptively.
🪟
Window Functions
ROW_NUMBER, RANK, LAG, LEAD — advanced analytics
Hard▾
📖 Definition
Window functions perform calculations across a set of rows related to the current row — without collapsing rows like GROUP BY does. Using OVER (PARTITION BY ... ORDER BY ...) syntax. Essential for ranking, running totals, moving averages, and row comparison.
window_functions.sql
-- ROW_NUMBER: unique rank per partition
SELECT name, dept, salary,
    ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS rank
FROM employees;

-- Get top earner per department
SELECT * FROM (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) rn
    FROM employees
) WHERE rn = 1;

-- Running total
SELECT date, amount,
    SUM(amount) OVER (ORDER BY date) AS running_total
FROM sales;

-- LAG/LEAD: compare with previous/next row
SELECT date, revenue,
    LAG(revenue, 1) OVER (ORDER BY date) AS prev_revenue,
    revenue - LAG(revenue, 1) OVER (ORDER BY date) AS growth
FROM monthly_revenue;
🎯 Interview Questions
QDifference between ROW_NUMBER, RANK, and DENSE_RANK?Medium▾
✅ Answer
For salaries [100, 90, 90, 80]: ROW_NUMBER: 1,2,3,4 (always unique, no ties). RANK: 1,2,2,4 (ties share rank, next rank skipped). DENSE_RANK: 1,2,2,3 (ties share rank, next rank NOT skipped). Use RANK for competition leaderboards. Use DENSE_RANK when you want no gaps. Use ROW_NUMBER for pagination.
🔒
Transactions, ACID & Isolation Levels
Database correctness guarantees
Hard▾
📖 Definition
ACID: Atomicity (all or nothing), Consistency (data remains valid), Isolation (concurrent transactions don't interfere), Durability (committed data survives crashes). Isolation levels control how much concurrent transactions see each other's uncommitted changes.
transactions.sql
-- Isolation levels (from weakest to strongest)
SET TRANSACTION ISOLATION LEVEL READ COMMITTED; -- PostgreSQL default

-- Problems by isolation level:
-- READ UNCOMMITTED: dirty reads (see uncommitted changes)
-- READ COMMITTED:   no dirty reads, but non-repeatable reads
-- REPEATABLE READ:  same rows each read, but phantom reads
-- SERIALIZABLE:     fully isolated, but slowest

-- Bank transfer — ACID example
BEGIN;
  UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
  UPDATE accounts SET balance = balance + 1000 WHERE id = 2;
COMMIT; -- both updates succeed, or ROLLBACK both
🎯 Interview Questions
QWhat is a dirty read, non-repeatable read, and phantom read?Hard▾
✅ Answer
Dirty read: reading data from a transaction that hasn't committed yet — that transaction might rollback, leaving you with invalid data. Non-repeatable read: read same row twice in a transaction, get different values (another transaction modified it between reads). Phantom read: run same query twice, get different number of rows (another transaction inserted/deleted rows between your reads).

🪝
React Hooks — useState, useEffect, useRef, useMemo
Hooks are the core of modern React
Medium▾
📖 Definition
Hooks are functions that let you use state and React features in functional components. useState: local component state. useEffect: side effects (API calls, subscriptions). useRef: mutable reference that doesn't trigger re-render. useMemo/useCallback: memoize expensive computations/functions.
HooksDemo.jsx
import { useState, useEffect, useRef, useMemo } from 'react';

function UserList({ departmentId }) {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const inputRef = useRef(null); // access DOM directly

  useEffect(() => {       // runs after render
    setLoading(true);
    fetchUsers(departmentId)
      .then(data => { setUsers(data); setLoading(false); });
    return () => { /* cleanup: cancel API call on unmount */ };
  }, [departmentId]); // dep array: re-run when departmentId changes

  // useMemo: only recalculate when users changes
  const activeUsers = useMemo(
    () => users.filter(u => u.active),
    [users]
  );

  if (loading) return <div>Loading...</div>;
  return <ul>{activeUsers.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}
🎯 Interview Questions
QWhat is the difference between useMemo and useCallback?Medium▾
✅ Answer
useMemo: memoizes the result of a function — returns a cached value. Use for expensive calculations. useCallback: memoizes the function itself — returns the same function reference. Use when passing callbacks to child components to prevent unnecessary re-renders.
const expensiveResult = useMemo(() => heavyCalc(data), [data]); const handleClick = useCallback(() => doSomething(id), [id]);
QWhen does useEffect run? Explain dependency array scenarios.Medium▾
✅ Answer
useEffect(fn) — no array: runs after EVERY render (rarely what you want). useEffect(fn, []) — empty array: runs ONCE after mount (componentDidMount equivalent). useEffect(fn, [dep1, dep2]) — runs after mount and whenever dep1 or dep2 changes. Return a cleanup function for subscriptions, timers, API cancellations.
QWhat is Virtual DOM and how does React's reconciliation work?Hard▾
✅ Answer
Virtual DOM is a lightweight JS copy of the real DOM. When state changes: (1) React creates a new Virtual DOM tree, (2) Diffing algorithm compares new vs old VDOM (O(n) using heuristics), (3) only the changed nodes get updated in the real DOM. This batching + minimal DOM updates is what makes React fast. The key prop helps React identify which list items changed.

🔄
Event Loop & Async
The most important JS concept to understand deeply
Hard▾
📖 Definition
JavaScript is single-threaded but handles async via the Event Loop. It has a Call Stack (executes code), Web APIs (setTimeout, fetch — handled by browser/Node), Microtask Queue (Promises — higher priority), and Macro-task Queue (setTimeout callbacks). Event loop constantly checks: if stack empty → drain microtasks → pick one macro-task.
EventLoop.js
console.log('1');                              // sync
setTimeout(() => console.log('2'), 0);       // macro-task
Promise.resolve().then(() => console.log('3')); // micro-task
console.log('4');                              // sync
// Output: 1, 4, 3, 2  (micro before macro!)

// async/await — syntactic sugar over Promises
async function loadUser(id) {
  try {
    const user = await fetch(`/api/users/${id}`).then(r => r.json());
    const orders = await fetch(`/api/orders/${user.id}`).then(r => r.json());
    return { user, orders };
  } catch (e) { console.error('Failed:', e); }
}

// Parallel — don't await sequentially when not dependent!
const [user, profile] = await Promise.all([fetchUser(), fetchProfile()]);
🎯 Interview Questions
QWhat are closures and give a real use case?Medium▾
✅ Answer
A closure is a function that retains access to its outer scope even after the outer function has returned. Real use: private counter, debounce, memoization, event handlers in loops.
function makeCounter() { let count = 0; // private — outer scope return { inc: () => ++count, get: () => count }; } const c = makeCounter(); c.inc(); c.inc(); c.get(); // 2 — count is private!
QExplain Debounce vs ThrottleMedium▾
✅ Answer
Debounce: waits until user STOPS calling for N ms, then fires once. Use for: search input (fire after typing stops), form validation, resize handler. Throttle: fires at most once per N ms regardless of how many times called. Use for: scroll events, mouse move, API rate limiting.

Prototype, 'this', and 'class'
JS OOP — prototypal inheritance under the hood
Hard▾
📖 Definition
Every JS object has a hidden [[Prototype]] link to another object. Property lookup walks this chain until found or null. 'this' depends on HOW a function is called, not where defined. Arrow functions don't have their own 'this' — they inherit from enclosing scope (lexical this).
Prototype.js
// 'this' depends on call context
const obj = {
  name: 'Sri',
  greet() { return `Hi ${this.name}`; },              // this = obj
  greetArrow: () => `Hi ${this.name}`,               // this = window!
  greetLater() { setTimeout(() => console.log(this.name), 100); } // arrow = outer this
};

// Prototype chain
function Animal(name) { this.name = name; }
Animal.prototype.speak = function() { return this.name; };
function Dog(name) { Animal.call(this, name); }
Dog.prototype = Object.create(Animal.prototype);
// ES6 class syntax is sugar over this:
class Dog extends Animal { bark() { return 'Woof!'; } }

02

Answering "What is your expected CTC?"

📝 Option 1 — Give a range (recommended)
"Based on my research and the responsibilities of this role, I'm looking at something in the range of [X–Y LPA]. That said, the overall package — learning opportunities, the team, and the role itself — matters to me too, and I'm open to discussing."
📝 Option 2 — Deflect (if asked early)
"I'd prefer to learn more about the full scope of the role before discussing compensation — I want to make sure it's a great fit for both of us first. Could you share the budgeted range for this position?"
03

Step-by-Step Negotiation Script

1
Get the offer in writing first
Never negotiate verbally without seeing the full offer letter. Ask: "Could you share the offer details in writing so I can review the full package?"
2
Take 24 hours — always
Say: "Thank you! I'm very excited. Could I have until tomorrow to review the package fully before responding?" This is standard and expected.
3
Counter with a specific number + reason
"Based on my 2.8 years of production experience with Java, Spring Boot, GCP, and CI/CD at Ford Pro, I was expecting closer to [X]. Is there flexibility?" — Always give a reason, not just a number.
4
If salary is fixed — negotiate benefits
Joining bonus, extra leave days, WFH flexibility, learning budget (Udemy/certs), earlier performance review (3 months instead of 6), better title. These cost less than salary hikes.
5
Accept gracefully — build the relationship
"I'm excited to accept and join the team. I appreciate the consideration and look forward to contributing." Your first impression post-offer matters for your long-term growth there.
